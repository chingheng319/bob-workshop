🤔 一般問題

Q1: Bob 和其他 AI 助手（如 ChatGPT、Copilot）有什麼不同？

✅ 答案

Bob 的獨特優勢：

  • 專案感知：Bob 理解整個專案結構，不只是單一檔案
  • 工具整合：12 個內建工具可直接操作檔案系統
  • 模式切換：Plan/Code/Advanced 模式適應不同需求
  • 可自訂：Rules、Skills、Slash Commands 完全客製化
  • 上下文管理：Memory Import 和智慧上下文載入

Q2: 什麼時候該用 Plan 模式？什麼時候用 Code 模式？

✅ 答案

模式 適用情境 範例
Plan 模式 • 需求分析
• 架構設計
• 問題診斷
• 文件撰寫 		"分析這個系統的架構"
"設計資料庫 Schema"
"規劃 API 端點"
Code 模式 • 程式碼實作
• Bug 修復
• 重構
• 測試撰寫 		"實作這個功能"
"修復這個 Bug"
"重構這段程式碼"

提示：可以隨時切換模式，使用 switch_mode 工具。

Q3: Bob 能取代人類開發者嗎？

⚠️ 答案

不能，也不應該。

Bob 是輔助工具，不是替代品：

  • AI 輔助：Bob 提供建議和產生程式碼
  • 人類決策：開發者負責最終決策和品質把關
  • 協作關係：Bob 處理重複性工作，人類專注創造性思考
  • 持續學習：開發者仍需理解程式碼和技術原理

重要：永遠要審查 Bob 產生的程式碼，確保正確性和安全性！

Q4: Bob 支援哪些程式語言？

✅ 答案

Bob 支援所有主流程式語言，包括但不限於：

  • 後端：Java, Python, JavaScript/TypeScript, Go, Rust, C#, PHP
  • 前端：React, Vue, Angular, HTML/CSS
  • 行動：Swift, Kotlin, React Native, Flutter
  • 資料：SQL, R, Julia
  • 其他：Shell Script, YAML, JSON, Markdown

🛠️ 工具使用問題

Q5: apply_diff 的 SEARCH 區塊一直匹配失敗怎麼辦？

⚠️ 常見原因與解決方案

原因 1：空白字元不一致

  • SEARCH 區塊必須完全匹配原始碼，包括空白和縮排
  • 解決：使用 read_file 重新讀取目標區域，確保完全一致

原因 2：沒有指定起始行號

  • 缺少 :start_line:N 標記
  • 解決：在 SEARCH 區塊第一行加上 :start_line:行號

原因 3：檔案已被修改

  • 檔案內容與 Bob 記憶中的不同
  • 解決：重新讀取檔案，使用最新內容

💡 最佳實踐

1. 使用 read_file 讀取目標區域
2. 複製完全相同的內容到 SEARCH 區塊
3. 加上 :start_line: 標記
4. 執行 apply_diff

Q6: 什麼時候用 write_to_file，什麼時候用 apply_diff？

✅ 選擇指南

使用 write_to_file：

  • ✅ 建立新檔案
  • ✅ 完全重寫小檔案（<100 行）
  • ✅ 配置檔案（JSON, YAML 等）

使用 apply_diff：

  • ✅ 修改現有程式碼
  • ✅ 大檔案的局部修改
  • ✅ 需要精確控制修改範圍
  • ✅ 批次修改多個位置

經驗法則：檔案 >100 行或只修改部分內容時，優先使用 apply_diff

💰 Token 與成本問題

Q8: 如何知道我用了多少 Token？

✅ 查看方式

  • Bob IDE 右下角顯示當前對話的 Token 使用量
  • 每次互動後會顯示消耗的 Bobcoin
  • 可在設定中查看總使用量和剩餘額度

Q9: Token 消耗太快，如何優化？

✅ 立即可行的優化策略

優先級 1：簡化回應（節省 50-70%）

  • 要求 Bob 保持回應簡潔
  • 避免冗長的解釋
  • 直接執行，不預告

優先級 2：使用行範圍（節省 80-95%）

  • 大檔案必用行範圍讀取
  • 只讀取需要的部分
  • 避免讀取整個檔案

優先級 3：批次操作（節省 40-60%）

  • 一次讀取多個檔案
  • 一次執行多個修改
  • 減少互動輪次

⚠️ 避免這些 Token 浪費

  • ❌ 重複讀取相同檔案
  • ❌ 讀取整個大檔案
  • ❌ 用 write_to_file 修改大檔案
  • ❌ 冗長的對話

Q10: Bobcoin 用完了怎麼辦？

✅ 解決方案

  • 聯繫管理員申請額外額度
  • 等待下個計費週期重置
  • 優化使用策略，減少消耗
  • 使用 Memory Import 減少重複載入

🔒 安全性問題

Q11: Bob 會儲存我的程式碼嗎？

⚠️ 重要資訊

根據 IBM Bob 的隱私政策：

  • 對話內容會暫時儲存以維持上下文
  • 不會用於訓練 AI 模型（需確認具體政策）
  • 建議不要分享敏感資訊
  • 使用 .bobignore 排除敏感檔案

安全原則：永遠假設 AI 可能看到你分享的內容，不要上傳真實的密碼、金鑰或個資！

Q12: 如何防止 Bob 讀取敏感檔案？

✅ 使用 .bobignore

步驟：

  1. 在專案根目錄建立 .bobignore 檔案
  2. 列出要排除的檔案和目錄
  3. 使用與 .gitignore 相同的語法

必須排除的內容：

  • 環境變數檔案（.env, .env.*）
  • 密鑰檔案（*.key, *.pem）
  • 配置檔案（含密碼的 config）
  • 資料庫檔案（*.db, *.sqlite）
  • 日誌檔案（*.log）

Q13: Bob 產生的程式碼安全嗎？

⚠️ 需要人工審查

Bob 產生的程式碼不保證完全安全，必須：

  • ✅ 使用 /review 或 /security-scan 檢查
  • ✅ 人工審查所有程式碼
  • ✅ 執行安全測試
  • ✅ 檢查輸入驗證
  • ✅ 確認無 SQL Injection、XSS 等漏洞

⚡ 效能問題

Q14: Bob 回應很慢怎麼辦？

✅ 可能原因與解決方案

原因 1：上下文過大

  • 解決：清理不需要的上下文，開始新對話

原因 2：讀取大量檔案

  • 解決：使用行範圍，只讀取必要部分

原因 3：複雜的任務

  • 解決：拆分成多個小任務

原因 4：網路問題

  • 解決：檢查網路連線，稍後重試

Q15: 如何加快 Bob 的回應速度？

✅ 優化技巧

  • 精確描述：清楚說明需求，避免來回確認
  • 減少上下文：只載入必要的檔案
  • 使用 Memory Import：預先載入常用資訊
  • 批次操作：一次完成多個任務
  • 簡化任務：複雜任務拆分成小步驟

✨ 最佳實踐總結

🎯 Bob 使用黃金法則

1. 規劃先行

  • 使用 Plan 模式先規劃，再用 Code 模式實作
  • 建立 AGENTS.md 指導開發方向
  • 撰寫 Custom Rules 確保品質

2. Token 優化

  • 大檔案必用行範圍讀取
  • 優先使用 apply_diff 而非 write_to_file
  • 批次操作減少互動輪次
  • 保持回應簡潔

3. 安全第一

  • 建立完整的 .bobignore
  • 不上傳真實敏感資料
  • 審查所有 AI 產生的程式碼
  • 使用 /security-scan 檢查安全性

4. 持續學習

  • 記錄有效的提示詞
  • 建立團隊知識庫
  • 分享最佳實踐
  • 定期更新 Rules 和 Skills

5. 人機協作

  • AI 輔助，人類決策
  • Bob 處理重複性工作
  • 人類專注創造性思考
  • 永遠保持批判性思維
