💡 開發流程建議

建議學員採用以下開發流程，充分運用 Bob 的 Plan 與 Code 模式：

1️⃣ 規劃階段 (Plan 模式)

  • 使用 Plan 模式分析需求與設計架構
  • 讓 Bob 協助規劃 Feedback Entity、Repository、Service、Controller 的設計
  • 確認資料庫結構與 API 端點設計
  • 產出開發計畫與實作步驟
  • 可請 Bob 產出系統架構圖、資料流程圖、ER 圖等視覺化文件

2️⃣ 開發階段 (Code 模式)

  • 切換至 Code 模式進行實際開發
  • 依照規劃逐步實作各層級元件
  • 使用 Bob 的程式碼生成與重構功能
  • 執行測試並驗證功能

3️⃣ 模式切換技巧

  • 遇到設計問題時，切回 Plan 模式重新規劃
  • 需要程式碼實作時，使用 Code 模式
  • 善用兩種模式的優勢，提升開發效率

💡 提示：Plan 模式適合思考與設計，Code 模式適合實作與除錯。靈活切換可以讓開發更有條理！

🎯 挑戰概述

組別任務

你/妳們是開發工程師，公司需要為現有的捷運車站管理系統新增乘客回饋功能，以收集乘客對車站設施、服務品質的意見，並支援營運改善決策。

你們將使用 IBM Bob 作為 AI 開發助手，從現有的 Sample Code 基礎上，共同快速實作這個使用者導向的回饋系統。

⚠️ 重要提醒

這是一個開放式挑戰，我們不會提供詳細的實作步驟。你需要：

  • 自行分析需求並設計解決方案
  • 善用 IBM Bob 的各種功能來加速開發

🎓 學習目標

完成本挑戰後，你/妳將能夠：

  • 使用 Bob 從現有程式碼快速新增功能
  • 建立自訂 Slash Command 提升開發效率
  • 撰寫專案 Rules 確保程式碼品質
  • 建立 AGENTS.md 指導 AI 助手
  • 實作完整的 CRUD 功能與前端整合

📋 功能需求

各分組實作以下核心功能：

📝 模組 1：回饋系統後端

功能說明：建立完整的回饋資料管理功能

  • ✅ 建立 Feedback Entity（回饋記錄）
  • ✅ 實作 FeedbackRepository 資料存取
  • ✅ 實作 FeedbackService 業務邏輯
  • ✅ 實作 FeedbackController REST API
  • ✅ 與 Station 建立關聯

要實作的 API 端點：

  • POST /api/feedbacks - 提交回饋
  • GET /api/feedbacks - 取得所有回饋
  • GET /api/feedbacks/station/{stationId} - 依車站查詢
  • GET /api/feedbacks/statistics - 回饋統計

🎨 模組 2：前端整合

功能說明：在現有 UI 中整合回饋功能

  • ✅ 回饋提交表單（車站選擇、評分、意見）
  • ✅ 回饋列表顯示
  • ✅ 依車站篩選回饋
  • ✅ 回饋統計圖表（平均評分、總數）

UI 設計重點：

  • 使用 Bootstrap 5 保持一致的設計風格
  • 即時更新，無需重新整理頁面
  • 友善的錯誤提示與成功訊息
  • 響應式設計，支援行動裝置

✅ 現有系統架構

Sample Code 已提供以下基礎功能：

元件 說明
Station Entity 車站資料（站名、代碼、路線）
StationRepository 車站資料存取介面
StationService 車站業務邏輯
StationController 車站 REST API（完整 CRUD）

🔌 待實作功能

你需要實作以下核心元件：

元件 說明
Feedback Entity 回饋記錄（車站、評分、意見、時間）
FeedbackRepository 回饋資料存取介面（含查詢方法）
FeedbackService 回饋業務邏輯（CRUD + 統計）
FeedbackController RESTful API 端點
前端整合 回饋表單、列表、統計顯示

💡 實作建議

建議實作順序

  1. 分析現有系統
    • 檢視現有的 Station Entity 與 Controller
    • 了解資料庫結構與 API 設計模式
    • 查看前端如何與後端互動
  2. 設計 Feedback Entity
    • 定義回饋記錄所需的欄位（評分、意見、時間等）
    • 建立與 Station 的關聯（ManyToOne）
    • 考慮資料驗證規則
  3. 建立 Repository 層
    • 建立 FeedbackRepository 介面
    • 設計查詢方法（依車站查詢、統計等）
  4. 實作 Service 層
    • 建立 FeedbackService 類別
    • 實作 CRUD 操作
    • 實作統計功能（平均評分、總數等）
  5. 建立 Controller 層
    • 建立 FeedbackController 類別
    • 實作 REST API 端點
    • 處理錯誤與驗證
  6. 前端整合
    • 在現有 UI 中加入回饋功能區塊
    • 實作表單提交與資料顯示
    • 加入統計圖表或資訊卡片
  7. 測試與驗證
    • 撰寫單元測試
    • 測試 API 端點
    • 驗證前端功能

技術考量重點：

  • 如何設計評分系統（1-5 星？數字評分？）
  • 如何確保回饋與車站的關聯正確？
  • 如何計算統計資料（平均評分、回饋數量）？
  • 如何設計友善的使用者介面？

🤖 產出要求

除了功能實作，你還需要建立以下 Bob 相關文件：

必要產出

  • 自訂 Slash Command：建立 /deploy-check 指令
    用途：讓 Bob 執行部署前檢查，驗證程式碼品質、測試覆蓋率、安全性等
  • 編碼規範：撰寫 .bob/rules/coding-standards.md
    內容：Spring Boot 開發規範、命名慣例、錯誤處理等
  • 安全規範：撰寫 .bob/rules/security-rules.md
    內容：輸入驗證、SQL Injection 防護、敏感資料處理等
  • AI 指引：建立 .bob/rules-code/AGENTS.md
    主題：捷運系統開發最佳實踐（資料驗證、API 設計、前端整合）

💡 提示

這些文件是幫助你更有效使用 Bob 的關鍵工具。Slash Command 讓 Bob 執行特定的部署檢查，Rules 確保程式碼品質，AGENTS.md 指導 AI 理解專案特性。

🚀 如何使用 Sample Application 完成挑戰

本專案提供完整的 Sample Application，包含基礎交易查詢功能。你需要在此基礎上新增警示監控功能。

步驟 1：Clone 團隊共用專案

⚠️ 重要：請 Clone 團隊成員 Fork 後的 Repository，而非原始專案

# 替換成團隊成員的 GitHub 帳號
git clone https://github.com/[團隊成員帳號]/bob-workshop.git
cd bob-workshop/topics/station-system

💡 確保所有團隊成員都從同一個 Fork 進行開發，才能順利協作

步驟 2：啟動應用程式

選擇以下任一方式：

方式 A：使用 Maven Wrapper（推薦）

./mvnw spring-boot:run

方式 B：使用已安裝的 Maven

mvn spring-boot:run

步驟 3：驗證啟動成功

看到以下訊息表示啟動成功：

Started StationApplication in X.XXX seconds

步驟 4：存取應用程式

  • Web UI: http://localhost:8080
  • H2 資料庫控制台: http://localhost:8080/h2-console
    • JDBC URL: jdbc:h2:mem:metro
    • Username: sa
    • Password: (留空)
  • API 端點: http://localhost:8080/api/stations

⚠️ 常見問題

Q: Port 8080 已被佔用？

A: 修改 src/main/resources/application.yml 中的 server.port

Q: Maven 下載依賴很慢？

A: 設定 Maven 使用國內鏡像源

Q: 如何停止應用程式？

A: 在終端機按 Ctrl + C

  • 開發者 A + B：
    • 建立 Feedback Entity 與 Repository
    • 實作 Service 層業務邏輯
    • 實作 Controller REST API
    • 與 Station 建立關聯

    • 顯示回饋列表與統計

  • 開發者 A + B：

    • 建立 Slash Commands (/deploy-check)
    • 建立 Custom Mode (Transit Security Expert)
  • 開發者 C：協助 PPT 成員整理程式碼截圖

  • 全員參與：
