💵 Bobcoin 計費機制

Token 與 Bobcoin 的關係

Bob 使用 Bobcoin 作為計費單位，與 Token 消耗直接相關：

項目 Token 消耗 Bobcoin 成本
輸入 Token 使用者訊息 + 系統提示 + 上下文 較低
輸出 Token Bob 的回應內容 較高（約 3-5 倍）
檔案讀取 檔案內容載入上下文 依檔案大小
工具呼叫 工具參數 + 執行結果 依複雜度

⚠️ 關鍵認知

輸出 Token 成本最高：Bob 的回應越長，成本越高。因此簡潔回應是節省成本的關鍵！

💡 成本優化原則

  • 優先級 1：減少輸出 Token（簡潔回應）
  • 優先級 2：減少檔案讀取 Token（使用行範圍）
  • 優先級 3：減少互動輪次（批次操作）
  • 優先級 4：避免重複載入（善用上下文）

📖 檔案讀取優化

策略 1：使用行範圍讀取

節省效果：80-95% Token 減少

❌ 低效方式：

讀取整個 5000 行的檔案
→ 消耗 ~5000 Token

✅ 高效方式：

只讀取目標函數所在的 234-267 行
→ 消耗 ~34 Token
→ 節省 99.3% Token！

💡 實作步驟

  1. 使用 list_code_definition_names 定位目標
  2. 使用 read_file 搭配
  3. 合併相近的行範圍（間隔 <10 行）

策略 2：批次讀取檔案

節省效果：減少 50% 互動輪次

❌ 低效方式：

read_file(file1) → 等待確認
read_file(file2) → 等待確認
read_file(file3) → 等待確認
→ 3 次互動輪次

✅ 高效方式：

read_file(file1, file2, file3) → 一次完成
→ 1 次互動輪次
→ 節省 66% 輪次！

💡 批次讀取範例



  src/app.ts
  
    src/utils.ts
    1-50
  
  src/config.ts

策略 3：避免重複讀取

節省效果：100% 避免浪費

⚠️ 常見錯誤

  • ❌ 重複讀取已在上下文中的檔案
  • ❌ 修改後立即重新讀取整個檔案
  • ❌ 不確定是否已讀取就再讀一次

✅ 正確做法

  • 記住已讀取的檔案內容
  • 使用 Memory Import 儲存重要資訊
  • 修改後直接進行下一步，不重讀

✏️ 程式碼編輯優化

策略 4：優先使用 apply_diff

節省效果：90%+ Token 減少（大檔案）

❌ 低效方式：

使用 write_to_file 重寫 1000 行檔案
只為了修改其中 10 行
→ 消耗 ~1000 Token

✅ 高效方式：

使用 apply_diff 只修改目標 10 行
→ 消耗 ~20 Token
→ 節省 98% Token！
檔案大小 write_to_file apply_diff 節省比例
100 行 ~100 Token ~20 Token 80%
500 行 ~500 Token ~20 Token 96%
1000 行 ~1000 Token ~20 Token 98%
5000 行 ~5000 Token ~20 Token 99.6%

策略 5：批次修改

節省效果：減少互動輪次

❌ 低效方式：

apply_diff 修改函數 A → 等待
apply_diff 修改函數 B → 等待
apply_diff 修改函數 C → 等待
→ 3 次互動

✅ 高效方式：

apply_diff 包含 3 個 SEARCH/REPLACE 區塊
→ 1 次互動完成所有修改
→ 節省 66% 輪次！

💬 回應簡潔性優化

策略 6：極簡回應

節省效果：50-70% 輸出 Token 減少

❌ 冗長回應（~150 Token）：

太好了！我很樂意幫助你更新這個配置檔案。
讓我先讀取檔案來了解目前的設定...

我已經讀取了檔案，現在我會將 API 端點從
http://old-api.com 更新為 http://new-api.com。
這個變更會影響應用程式連接後端服務的方式...

更新完成！我已經成功修改了配置檔案。

✅ 簡潔回應（~10 Token）：

[直接使用 read_file 工具]
[直接使用 apply_diff 工具]
已更新 API 端點

⚠️ 要避免的冗詞

  • ❌ "太好了！"、"當然！"、"我很樂意..."
  • ❌ "讓我先..."、"現在我會..."
  • ❌ 重複使用者的問題
  • ❌ 解釋明顯的步驟
  • ❌ 不必要的總結

✅ 簡潔原則

  • 直接執行，不預告
  • 用工具說話，少用文字
  • 結果導向，不解釋過程
  • 使用項目符號，不用段落

策略 7：讓程式碼自己說話

節省效果：避免重複說明

❌ 過度說明：

我已經新增了一個新的函數 calculateTotal，
這個函數接收一個 items 陣列作為參數，
然後使用 reduce 方法來計算總和，
最後回傳計算結果。這樣可以讓程式碼更簡潔...

✅ 簡潔說明：

新增 calculateTotal 函數

原則：程式碼本身就是最好的說明，不需要重複解釋明顯的邏輯。

🔄 批次操作策略

策略 8：規劃後執行

節省效果：減少 40-60% 互動輪次

❌ 逐步執行：

1. 讀取 file1 → 等待確認
2. 修改 file1 → 等待確認
3. 讀取 file2 → 等待確認
4. 修改 file2 → 等待確認
5. 執行測試 → 等待確認
→ 5 次互動輪次

✅ 批次執行：

1. 批次讀取 file1, file2 → 等待確認
2. 批次修改 file1, file2 → 等待確認
3. 執行測試 → 完成
→ 3 次互動輪次
→ 節省 40% 輪次！

💡 批次操作清單

  • ✅ 一次讀取多個檔案（最多 10 個）
  • ✅ 一次 apply_diff 包含多個修改
  • ✅ 規劃完整後再執行
  • ✅ 避免來回確認

🧠 上下文管理

策略 9：善用 Memory Import

節省效果：避免重複載入常用資訊

💡 適合 Memory Import 的內容

  • 專案架構說明
  • API 規格文件
  • 編碼規範
  • 常用程式碼片段
  • 資料庫 Schema

⚠️ 不適合 Memory Import 的內容

  • ❌ 完整的程式碼檔案（用 read_file）
  • ❌ 會頻繁變動的內容
  • ❌ 過於龐大的文件

策略 10：清理不需要的上下文

節省效果：降低每次互動的基礎成本

💡 上下文清理時機

  • 完成一個大任務後
  • 切換到不相關的任務時
  • 上下文累積過多時
  • 開始新的開發階段時

✨ 最佳實踐總結

🎯 Token 優化黃金法則

  1. 簡潔至上：輸出 Token 成本最高，極簡回應是關鍵
  2. 精確讀取：大檔案必用行範圍，可節省 80-95% Token
  3. 批次操作：一次完成多個操作，減少互動輪次
  4. 工具優先：apply_diff 優於 write_to_file
  5. 避免重複：不重讀已載入的檔案
  6. 規劃執行：先規劃完整方案，再批次執行
  7. 善用記憶：常用資訊存入 Memory Import
  8. 定期清理：清理不需要的上下文

⚠️ 成本陷阱

  • ❌ 冗長的回應（輸出 Token 成本高）
  • ❌ 讀取整個大檔案
  • ❌ 用 write_to_file 修改大檔案
  • ❌ 重複讀取相同檔案
  • ❌ 逐步執行可批次的操作
