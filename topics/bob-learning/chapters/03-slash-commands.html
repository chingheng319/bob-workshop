⚡ 3.1 什麼是 Slash Commands？

定義

Slash Commands（斜線指令）是 Bob 提供的快捷指令系統，讓你用簡短的指令快速執行常見任務。

使用方式

在對話框中輸入 / 開頭的指令，例如：

/help

💡 可在 Bob Shell 或 Bob IDE 中使用

⚠️ 重要：不同環境的指令差異

Bob Shell 和 Bob IDE 支援的內建指令數量不同：

🖥️ Bob Shell（28 個內建指令）

  • 完整的 28 個起始內建 Slash Commands
  • 包含指令：
    about, bug, chat, clear, compress, copy, create-pr, directory, docs, editor, feedback, help, ide, init, instance, mcp, memory, mode, permissions, quit, review, secrets-manager, settings, stats, terminal-setup, theme, tools, vim
  • 支援建立自訂指令

💻 Bob IDE（4 個內建指令）

  • 支援 4 個起始內建指令：
    • init - 分析程式碼庫並為 AI 助理建立簡潔的 AGENTS.md 檔案
    • create-pr - 建立拉取請求
    • review - 審查程式碼變更
    • start_java_modernization - 啟動 Java 現代化工作流程
  • 支援建立自訂指令

💡 重點：如果你需要使用更多內建指令（如 /help、/doc、/test 等），請使用 Bob Shell。Bob IDE 主要專注於程式碼開發相關的核心指令。

💡 為什麼要使用 Slash Commands？

  • 提升效率：一個指令完成複雜任務
  • 標準化：團隊使用統一的指令
  • 可重複：將常用操作封裝成指令
  • 易記憶：簡短的指令名稱

📋 3.2 內建指令介紹

Bob IDE 的 4 個內建指令

指令 用途 使用場景
/init 建立 AGENTS.md 分析專案並建立 AI 助理指引文件
/create-pr 建立拉取請求 從當前分支建立 PR
/review 程式碼審查 審查程式碼變更（可指定分支）
/start_java_modernization Java 現代化 啟動 Java 程式碼現代化流程

Bob Shell 的常用指令（28 個）

指令 用途 使用場景
/help 顯示幫助資訊 查看所有可用指令
/clear 清除對話歷史 開始新任務時重置 Context
/init 建立 AGENTS.md 分析專案並建立 AI 助理指引
/review 程式碼審查 審查程式碼變更
/create-pr 建立拉取請求 從當前分支建立 PR
/memory 記憶體管理 管理對話記憶體
/docs 開啟文件 在瀏覽器中開啟 Bob Shell 文件
...以及其他 21 個指令（theme, tools, settings, vim, mode, copy 等）

💡 如何查看完整指令列表

在 Bob Shell 中：輸入 /help 查看所有 28 個指令

在 Bob IDE 中：輸入 / 會自動顯示 4 個可用指令

💡 使用範例

範例 1：審查程式碼（Bob IDE 和 Bob Shell 都支援）

/review

審查當前分支的程式碼變更

範例 2：建立 AGENTS.md（Bob IDE 和 Bob Shell 都支援）

/init

分析專案並建立 AGENTS.md 檔案

範例 3：清除對話（僅 Bob Shell）

/clear

清除對話歷史，重新開始

🛠️ 3.3 建立自訂 Slash Commands

為什麼要建立自訂指令？

自訂指令讓你可以：

  • 封裝專案特定的操作
  • 標準化團隊的工作流程
  • 提升重複任務的效率

建立步驟

  1. 建立指令檔案

    在專案根目錄建立 .bob/commands/ 目錄

  2. 定義指令

    建立 指令名稱.md 檔案

  3. 撰寫指令內容

    使用 Markdown 格式描述指令行為

📝 實戰範例：建立 /security-scan 指令

目標：建立一個安全掃描指令，檢查程式碼中的安全問題

步驟 1：建立目錄

mkdir -p .bob/commands

步驟 2：建立指令檔案

touch .bob/commands/security-scan.md

步驟 3：撰寫指令內容

# Security Scan Command

## 目的
執行程式碼安全掃描，檢查常見的安全漏洞

## 檢查項目
1. SQL Injection 風險
2. XSS 漏洞
3. 敏感資料外洩
4. 不安全的加密方式
5. 輸入驗證問題

## 執行步驟
1. 讀取指定的檔案或目錄
2. 分析程式碼中的安全問題
3. 產生詳細的安全報告
4. 提供修復建議

## 使用方式
```
/security-scan [檔案路徑]
```

## 範例
```
/security-scan src/main/java/com/example/UserController.java
```

✏️ 3.4 實戰練習

練習 1：使用內建指令

在 Bob IDE 中（4 個指令）：

  1. 使用 /init 為專案建立 AGENTS.md
  2. 使用 /review 審查程式碼變更
  3. 使用 /create-pr 建立拉取請求

在 Bob Shell 中（28 個指令）：

  1. 使用 /help 查看所有可用指令
  2. 使用 /clear 清除對話歷史
  3. 使用 /memory 管理記憶體
  4. 使用 /docs 開啟文件

練習 2：建立自訂指令

請建立一個 /api-doc 指令，用於產生 API 文件：

  1. 建立 .bob/commands/api-doc.md
  2. 定義指令的目的與執行步驟
  3. 測試指令是否正常運作（Bob Shell 和 Bob IDE 都支援）

練習 3：團隊指令

思考你的團隊常做的任務，設計 2-3 個自訂指令：

  • 例如：/deploy-check（部署前檢查）
  • 例如：/pr-review（Pull Request 審查）
  • 例如：/performance-check（效能檢查）

💡 3.5 Slash Commands 最佳實踐

指令命名原則

  • 簡短明確：使用簡短但有意義的名稱
  • 動詞開頭：如 /check、/generate
  • 連字符分隔：多個單字用 - 連接
  • 避免衝突：不要與內建指令重名

指令文件撰寫

  • 清楚的目的說明：讓使用者知道指令的用途
  • 詳細的執行步驟：Bob 需要知道如何執行
  • 使用範例：提供具體的使用案例
  • 參數說明：如果有參數，要清楚說明

團隊協作

  • 版本控制：將指令檔案加入 Git
  • 文件化：在 README 中列出可用指令
  • 定期審查：檢查指令是否仍然有用
  • 團隊共識：確保團隊成員都了解指令用途
