📋 4.1 什麼是 Custom Rules？

定義

Custom Rules（自訂規則）是你為專案建立的規範文件，告訴 Bob 在開發時應該遵循哪些規則、慣例和最佳實踐。

為什麼需要 Custom Rules？

  • 統一規範：確保團隊成員遵循相同的編碼標準
  • 提升品質：讓 Bob 產生符合專案要求的程式碼
  • 減少審查：自動遵循規則，減少人工審查時間
  • 知識傳承：將團隊經驗固化為規則文件

💡 實際效益

有了 Custom Rules，Bob 會：

  • 使用專案指定的命名慣例
  • 遵循特定的程式碼風格
  • 套用安全最佳實踐
  • 符合效能優化要求
  • 產生符合規範的文件與註解

📁 4.2 Rules 檔案結構

檔案位置

Custom Rules 檔案放在專案根目錄的 .bob/rules/ 目錄下：

專案根目錄/
├── .bob/
│   └── rules/
│       ├── coding-standards.md
│       ├── security-rules.md
│       ├── naming-conventions.md
│       └── performance-guidelines.md
└── src/

檔案格式

  • 格式：Markdown (.md)
  • 編碼：UTF-8
  • 命名：使用描述性名稱，小寫加連字符
  • 組織：可依主題分成多個檔案

💡 建議的規則分類

  • coding-standards.md - 編碼規範
  • security-rules.md - 安全規則
  • naming-conventions.md - 命名慣例
  • api-design.md - API 設計原則
  • testing-guidelines.md - 測試指南
  • documentation-standards.md - 文件標準

✍️ 4.3 撰寫有效的 Rules

規則撰寫原則

  1. 明確具體

    ❌ "程式碼要寫得好"
    ✅ "類別名稱使用 PascalCase，方法名稱使用 camelCase"

  2. 提供範例

    包含正確與錯誤的程式碼範例

  3. 說明原因

    解釋為什麼要遵循這個規則

  4. 可執行性

    規則要能被 Bob 理解並執行

📝 規則文件範本

# 規則標題

## 目的
說明這個規則的目的與重要性

## 規則內容
具體的規則描述

## 正確範例
```java
// 正確的寫法
public class UserService {
    private final UserRepository userRepository;
    
    public User findById(Long id) {
        return userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
    }
}
```

## 錯誤範例
```java
// 錯誤的寫法
public class userservice {
    public User FindById(long ID) {
        return repo.find(ID);
    }
}
```

## 例外情況
說明何時可以不遵循此規則

🎯 4.4 實戰範例：建立編碼規範

範例：coding-standards.md

# 編碼規範

## 命名慣例

### 類別命名
- 使用 PascalCase
- 名稱應該是名詞
- 避免縮寫（除非是廣為人知的）

**正確範例**：
```java
public class UserService { }
public class OrderRepository { }
```

**錯誤範例**：
```java
public class userService { }  // 應使用 PascalCase
public class UsrSvc { }       // 避免縮寫
```

### 方法命名
- 使用 camelCase
- 名稱應該是動詞或動詞片語
- Boolean 方法使用 is/has/can 開頭

**正確範例**：
```java
public User findById(Long id) { }
public boolean isActive() { }
public boolean hasPermission() { }
```

### 變數命名
- 使用 camelCase
- 名稱應該有意義且描述性
- 避免單字母變數（除了迴圈計數器）

**正確範例**：
```java
String userName = "John";
int totalCount = 0;
List activeUsers = new ArrayList<>();
```

## 程式碼組織

### 類別結構順序
1. 靜態變數
2. 實例變數
3. 建構子
4. 公開方法
5. 私有方法

### 方法長度
- 單一方法不超過 50 行
- 如果超過，考慮拆分成多個小方法

## 註解規範

### JavaDoc
- 所有公開類別和方法都要有 JavaDoc
- 說明參數、回傳值、可能的例外

**範例**：
```java
/**
 * 根據 ID 查找使用者
 * 
 * @param id 使用者 ID
 * @return 使用者物件
 * @throws UserNotFoundException 當使用者不存在時
 */
public User findById(Long id) {
    // 實作
}
```

## 錯誤處理

### 例外處理原則
- 不要捕捉通用的 Exception
- 使用特定的例外類型
- 記錄例外資訊

**正確範例**：
```java
try {
    // 操作
} catch (IOException e) {
    log.error("檔案讀取失敗", e);
    throw new FileProcessingException("無法處理檔案", e);
}
```

**錯誤範例**：
```java
try {
    // 操作
} catch (Exception e) {  // 太廣泛
    // 空的 catch block
}
```

🔒 4.5 實戰範例：安全規則

範例：security-rules.md

# 安全規則

## SQL Injection 防護

### 規則
永遠使用參數化查詢，不要拼接 SQL 字串

**正確範例**：
```java
String sql = "SELECT * FROM users WHERE id = ?";
PreparedStatement stmt = conn.prepareStatement(sql);
stmt.setLong(1, userId);
```

**錯誤範例**：
```java
String sql = "SELECT * FROM users WHERE id = " + userId;  // 危險！
```

## 密碼處理

### 規則
- 使用 BCrypt 或 Argon2 加密密碼
- 不要在日誌中記錄密碼
- 密碼長度至少 8 個字元

**正確範例**：
```java
String hashedPassword = BCrypt.hashpw(plainPassword, BCrypt.gensalt());
```

## 輸入驗證

### 規則
所有使用者輸入都必須驗證

**正確範例**：
```java
@PostMapping("/users")
public User createUser(@Valid @RequestBody UserDto userDto) {
    // @Valid 會自動驗證
}
```

## 敏感資料

### 規則
- 不要在程式碼中硬編碼密鑰、密碼
- 使用環境變數或密鑰管理服務
- 敏感資料要加密儲存

**正確範例**：
```java
String apiKey = System.getenv("API_KEY");
```

**錯誤範例**：
```java
String apiKey = "sk-1234567890";  // 不要硬編碼！
```

✏️ 4.6 實戰練習

練習 1：建立基本規則檔案

為你的專案建立第一個規則檔案：

  1. 建立 .bob/rules/ 目錄
  2. 建立 coding-standards.md
  3. 撰寫至少 3 條編碼規範
  4. 包含正確與錯誤範例

練習 2：測試規則效果

驗證 Bob 是否遵循你的規則：

  1. 請 Bob 建立一個新的類別
  2. 檢查是否符合你定義的命名慣例
  3. 檢查程式碼結構是否符合規範
  4. 如果不符合，調整規則描述使其更明確

練習 3：建立安全規則

為專案建立安全規則：

  1. 建立 security-rules.md
  2. 定義 SQL Injection 防護規則
  3. 定義密碼處理規則
  4. 定義輸入驗證規則

💡 4.7 Custom Rules 最佳實踐

規則撰寫技巧

  • 從簡單開始：先建立核心規則，逐步擴充
  • 保持更新：隨專案演進更新規則
  • 團隊共識：規則應該是團隊討論的結果
  • 定期審查：檢查規則是否仍然適用
  • 提供範例：每條規則都要有程式碼範例

規則組織建議

  • 分類清楚：依主題分成不同檔案
  • 優先順序：重要規則放在前面
  • 交叉引用：相關規則互相連結
  • 版本控制：規則檔案加入 Git

⚠️ 常見錯誤

  • ❌ 規則太模糊，Bob 無法理解
  • ❌ 規則太多太複雜，難以遵循
  • ❌ 沒有提供範例，只有文字描述
  • ❌ 規則互相矛盾
  • ❌ 從不更新規則檔案
