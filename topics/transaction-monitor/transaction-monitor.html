💡 開發流程建議

建議學員採用以下開發流程，充分運用 Bob 的 Plan 與 Code 模式：

1️⃣ 規劃階段 (Plan 模式)

  • 使用 Plan 模式分析需求與設計架構
  • 讓 Bob 協助規劃 Entity、Repository、Service、Controller 的設計
  • 確認資料庫結構與 API 端點設計
  • 產出開發計畫與實作步驟
  • 可請 Bob 產出系統架構圖、資料流程圖、ER 圖等視覺化文件

2️⃣ 開發階段 (Code 模式)

  • 切換至 Code 模式進行實際開發
  • 依照規劃逐步實作各層級元件
  • 使用 Bob 的程式碼生成與重構功能
  • 執行測試並驗證功能

3️⃣ 模式切換技巧

  • 遇到設計問題時，切回 Plan 模式重新規劃
  • 需要程式碼實作時，使用 Code 模式
  • 善用兩種模式的優勢，提升開發效率

💡 提示：Plan 模式適合思考與設計，Code 模式適合實作與除錯。靈活切換可以讓開發更有條理！

🎯 挑戰概述

你的任務

你是資深開發工程師，公司需要為現有的信用卡交易系統新增即時交易監控功能，以識別異常交易模式並提供風險預警。

你將使用 IBM Bob 作為 AI 開發助手，從現有的 Sample Code 基礎上，快速實作這個關鍵的安全功能。

⚠️ 重要提醒

這是一個開放式挑戰，我們不會提供詳細的實作步驟。你需要：

  • 自行分析需求並設計解決方案
  • 善用 IBM Bob 的各種功能來加速開發

🎓 學習目標

完成本挑戰後，你將能夠：

  • 使用 Bob 從現有程式碼快速新增功能
  • 建立自訂 Slash Command 提升開發效率
  • 撰寫專案 Rules 確保程式碼品質
  • 建立 AGENTS.md 指導 AI 助手
  • 實作金融系統的安全最佳實踐

📋 功能需求

你需要實作以下 2 個核心功能模組：

🚨 模組 1：異常偵測引擎

功能說明：針對現有交易資料進行異常模式偵測

  • ✅ 高額交易偵測：單筆金額 > 50,000 元
  • ✅ 頻繁交易偵測：1 小時內同一卡片 > 5 筆交易
  • ✅ 重複交易偵測：5 分鐘內相同卡片、商店、金額

實作方式：

  • 使用現有 TransactionService 查詢方法
  • 分析交易資料（data.sql 或新增的交易）
  • 產生 TransactionAlert 警示記錄

📊 模組 2：警示管理系統

功能說明：管理偵測到的異常交易警示

  • 🔍 警示查詢：依狀態、風險等級、時間範圍篩選
  • 📈 統計報表：警示數量、風險分布、處理狀態
  • ✅ 警示處理：標記為已處理/誤報

實作方式：

  • 建立 AlertRepository 和 AlertService
  • 實作 RESTful API 端點
  • 提供查詢、統計、處理功能

💡 實作方案選擇

方案 A：手動偵測（簡化版）

  • 透過 POST /api/alerts/detect 手動觸發偵測
  • 分析 data.sql 中的現有交易資料
  • 適合 Workshop 快速展示

方案 B：自動偵測（完整版）⭐ 推薦

  • 實作 POST /api/transactions 新增交易 API
  • 每次新增交易時自動觸發異常偵測
  • 更貼近真實業務場景
  • 可透過 API 動態新增測試資料

建議：先實作方案 A 驗證偵測邏輯，再擴充方案 B 完善功能

❓ 為什麼兩個模組不能並行開發？

本專案的兩個模組有共同的資料層依賴：

🔍 依賴分析

  • 模組 1（異常偵測引擎）：分析現有 Transaction 資料，產生 TransactionAlert 警示記錄
  • 模組 2（警示管理系統）：管理已產生的 TransactionAlert（查詢、統計、處理）
  • 兩個模組都需要操作 TransactionAlert Entity 和 AlertRepository

📊 開發順序

Person A 建立資料層
├─ TransactionAlert Entity
└─ AlertRepository
        ↓
Person B 實作業務邏輯
├─ AlertDetectionService（偵測邏輯）
└─ AlertController（管理 API）

💡 重點：Person A 必須先完成 TransactionAlert Entity 和 AlertRepository，Person B 才能開始實作偵測與管理功能。
