📖 6.1 什麼是 AGENTS.md？

定義

AGENTS.md 是專案的「AI 助理指引文件」，告訴 Bob 關於你的專案的重要資訊，包括架構、技術棧、開發流程、注意事項等。

為什麼需要 AGENTS.md？

  • 快速理解專案：Bob 能快速掌握專案全貌
  • 減少錯誤：避免 Bob 做出不符合專案的決策
  • 提升效率：不需要每次都重複說明專案背景
  • 知識傳承：新成員（包括 AI）能快速上手

💡 AGENTS.md vs README.md

README.md：給人類看的專案說明

  • 專案介紹、安裝步驟、使用方式
  • 面向使用者或貢獻者

AGENTS.md：給 AI 助理看的專案指引

  • 架構設計、技術決策、開發規範
  • 面向 AI 開發助手
  • 更技術性、更結構化

📁 6.2 AGENTS.md 檔案結構

檔案位置

AGENTS.md 放在專案根目錄的 .bob/ 目錄下：

專案根目錄/
├── .bob/
│   ├── AGENTS.md          ← 主要的 AI 指引文件
│   ├── rules/
│   ├── skills/
│   └── commands/
├── README.md
└── src/

建議的章節結構

  1. 專案概述：專案目的、核心功能
  2. 技術棧：使用的技術與框架
  3. 架構設計：系統架構、模組劃分
  4. 目錄結構：專案檔案組織方式
  5. 開發流程：如何開發新功能
  6. 測試策略：測試方法與工具
  7. 部署流程：如何部署應用
  8. 注意事項：重要的限制或特殊要求

🚀 6.3 使用 /init 自動產生 AGENTS.md

最簡單的方式

Bob 提供 /init 指令，可以自動分析你的專案並產生 AGENTS.md：

/init

Bob 會：

  • 分析專案的檔案結構
  • 識別使用的技術棧
  • 理解專案架構
  • 產生簡潔的 AGENTS.md

⚠️ 使用提示

  • /init 在 Bob IDE 和 Bob Shell 都可以使用
  • 產生後可以手動編輯補充細節
  • 建議在專案初期就建立 AGENTS.md
  • 隨專案演進定期更新內容

📝 6.4 AGENTS.md 範本

完整範本

# 專案名稱

## 專案概述
簡要說明專案的目的、核心功能和業務價值

## 技術棧
- **後端**: Spring Boot 3.5, Java 17
- **資料庫**: PostgreSQL 15
- **快取**: Redis
- **訊息佇列**: RabbitMQ
- **測試**: JUnit 5, Mockito
- **建置工具**: Maven

## 架構設計

### 整體架構
採用分層架構：
- Controller 層：處理 HTTP 請求
- Service 層：業務邏輯
- Repository 層：資料存取
- Model 層：資料模型

### 核心模組
1. **使用者模組** (`com.example.user`)
   - 使用者註冊、登入、權限管理
   
2. **訂單模組** (`com.example.order`)
   - 訂單建立、查詢、狀態管理
   
3. **支付模組** (`com.example.payment`)
   - 支付處理、退款、對帳

## 目錄結構
```
src/
├── main/
│   ├── java/
│   │   └── com/example/
│   │       ├── controller/    # REST API 端點
│   │       ├── service/       # 業務邏輯
│   │       ├── repository/    # 資料存取
│   │       ├── model/         # 資料模型
│   │       ├── dto/           # 資料傳輸物件
│   │       ├── config/        # 配置類別
│   │       └── exception/     # 例外處理
│   └── resources/
│       ├── application.yml    # 應用配置
│       ├── data.sql          # 測試資料
│       └── schema.sql        # 資料庫結構
└── test/                     # 測試程式碼
```

## 開發流程

### 新增功能
1. 在 Service 層實作業務邏輯
2. 在 Repository 層新增資料存取方法
3. 在 Controller 層建立 API 端點
4. 撰寫單元測試
5. 更新 API 文件

### 命名慣例
- 類別：PascalCase
- 方法：camelCase
- 常數：UPPER_SNAKE_CASE
- 套件：小寫，單字間用點分隔

## 測試策略
- 單元測試覆蓋率目標：80%
- 使用 JUnit 5 + Mockito
- 整合測試使用 TestContainers
- API 測試使用 RestAssured

## 資料庫設計原則
- 使用 Flyway 管理資料庫遷移
- 所有表格都要有 created_at 和 updated_at
- 使用 UUID 作為主鍵
- 敏感資料要加密儲存

## API 設計規範
- RESTful 風格
- 使用 HTTP 狀態碼表示結果
- 統一的錯誤回應格式
- 所有 API 都要有版本號 (v1, v2...)

## 安全要求
- 所有 API 都要驗證 JWT Token
- 敏感操作要記錄審計日誌
- 輸入資料都要驗證
- 使用參數化查詢防止 SQL Injection

## 效能考量
- 查詢結果要分頁
- 使用 Redis 快取熱門資料
- 避免 N+1 查詢問題
- 大量資料處理使用非同步

## 部署流程
1. 執行測試：`mvn test`
2. 建置：`mvn clean package`
3. Docker 映像：`docker build -t app:latest .`
4. 部署到 Kubernetes

## 注意事項
- 不要在程式碼中硬編碼密鑰
- 所有外部 API 呼叫都要有重試機制
- 記得更新 CHANGELOG.md
- 重大變更要通知團隊

## 相關文件
- [API 文件](docs/api/)
- [資料庫設計](docs/database/)
- [部署指南](docs/deployment/)

- 專案經理：Jane Smith (jane@example.com)

🎯 6.5 實戰範例：Spring Boot 專案

範例：電商系統的 AGENTS.md

# 電商系統

## 專案概述
這是一個 B2C 電商平台，提供商品瀏覽、購物車、訂單處理、支付等功能。

## 技術棧
- Spring Boot 3.5
- Spring Security (JWT 認證)
- Spring Data JPA
- PostgreSQL
- Redis (Session & Cache)
- RabbitMQ (非同步處理)

## 核心業務流程

### 下單流程
1. 使用者將商品加入購物車
2. 確認訂單資訊
3. 選擇支付方式
4. 呼叫支付 API
5. 更新訂單狀態
6. 發送訂單確認郵件（非同步）

### 庫存管理
- 使用樂觀鎖防止超賣
- 下單時預留庫存
- 支付成功後扣除庫存
- 訂單取消時釋放庫存

## 重要的設計決策

### 為什麼使用 Redis？
- Session 管理：支援分散式部署
- 快取：減少資料庫查詢
- 分散式鎖：防止併發問題

### 為什麼使用 RabbitMQ？
- 非同步處理：郵件發送、日誌記錄
- 解耦：訂單系統與通知系統分離
- 削峰：處理高併發訂單

## 開發注意事項

### 交易處理
- 所有涉及金額的操作都要使用 @Transactional
- 使用 BigDecimal 處理金額，不要用 double
- 記錄所有交易日誌

### 併發控制
- 庫存扣減使用樂觀鎖
- 訂單狀態更新使用悲觀鎖
- 避免長時間持有鎖

### 錯誤處理
- 支付失敗要回滾庫存
- 外部 API 呼叫要有重試機制
- 所有錯誤都要記錄日誌

## 測試要求
- Controller 層：測試 API 端點
- Service 層：測試業務邏輯
- Repository 層：測試資料存取
- 整合測試：測試完整流程

## 效能指標
- API 回應時間 < 200ms
- 資料庫查詢 < 100ms
- 快取命中率 > 80%
- 併發處理 > 1000 TPS

✏️ 6.6 實戰練習

練習 1：使用 /init 產生 AGENTS.md

為你的專案自動產生 AGENTS.md：

  1. 在專案根目錄執行 /init
  2. 檢查產生的 AGENTS.md 內容
  3. 補充 Bob 無法自動識別的資訊
  4. 加入重要的設計決策說明

練習 2：手動撰寫 AGENTS.md

如果專案較複雜，手動撰寫更詳細的文件：

  1. 使用範本建立 .bob/AGENTS.md
  2. 填寫專案概述與技術棧
  3. 描述架構設計與核心模組
  4. 記錄重要的開發注意事項

練習 3：測試 AGENTS.md 效果

驗證 Bob 是否理解你的專案：

  1. 請 Bob 說明專案的架構
  2. 請 Bob 新增一個符合規範的功能
  3. 檢查 Bob 是否遵循 AGENTS.md 的指引
  4. 根據結果調整文件內容

💡 6.7 AGENTS.md 最佳實踐

撰寫技巧

  • 簡潔明確：避免冗長的描述
  • 結構化：使用清楚的章節劃分
  • 重點突出：強調重要的設計決策
  • 實例說明：提供具體的程式碼範例
  • 持續更新：隨專案演進更新文件

內容建議

  • 技術決策：說明為什麼選擇某個技術
  • 限制條件：記錄已知的限制或問題
  • 最佳實踐：團隊認可的開發方式
  • 常見陷阱：容易犯的錯誤
  • 參考資源：相關文件的連結

⚠️ 常見錯誤

  • ❌ 內容太簡略，Bob 無法理解專案
  • ❌ 內容太冗長，重點不明確
  • ❌ 只描述「是什麼」，沒說明「為什麼」
  • ❌ 文件過時，與實際程式碼不符
  • ❌ 缺少具體範例
