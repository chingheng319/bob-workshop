## 💬 乘客回饋系統（核心功能）

### 🎯 設計目標

#### 1. 建立雙向溝通橋樑
- **傳統問題**：乘客意見無處反映
- **解決方案**：提供便捷的線上回饋管道
- **預期效果**：提升服務品質、增強乘客滿意度

#### 2. 數據驅動決策
- **收集結構化回饋**：評分、類別、內容
- **分析服務痛點**：找出需改善的車站或設施
- **追蹤處理進度**：待處理 → 處理中 → 已完成

### 📝 功能特色

#### **提交回饋表單**
```
必填欄位：
├─ 車站選擇（下拉選單）
├─ 姓名
├─ Email
├─ 評分（1-5星）
├─ 類別（服務品質/設施狀況/清潔度/安全性/其他）
└─ 回饋內容（最多1000字）
```

**設計考量：**
- ✅ 簡潔明瞭的表單設計
- ✅ 必填欄位清楚標示
- ✅ 下拉選單減少輸入錯誤
- ✅ 星級評分直觀易懂
- ✅ 分類管理便於後續分析

#### **回饋列表展示**
```
顯示資訊：
├─ 乘客姓名
├─ 車站名稱
├─ 提交時間
├─ 星級評分（視覺化）
├─ 回饋類別（彩色徽章）
├─ 處理狀態（待處理/處理中/已完成）
├─ 回饋內容
└─ 管理員回覆（如有）
```

**設計考量：**
- ✅ 最新10筆回饋優先顯示
- ✅ 顏色編碼快速識別評分
- ✅ 狀態標籤追蹤處理進度
- ✅ 捲軸設計節省空間
- ✅ 管理員回覆增加透明度

## 🎨 設計理念

### 核心原則

#### 1. 使用者中心設計 (User-Centered Design)
```
設計流程：
了解需求 → 設計方案 → 實作測試 → 持續優化
```

**實踐方式：**
- 分析使用者痛點（無回饋管道）
- 設計簡潔表單（降低使用門檻）
- 提供即時回饋（增強互動感）

#### 2. 資訊架構優化 (Information Architecture)
```
組織原則：
分類清晰 → 層次分明 → 易於導航
```

**實踐方式：**
- 採用分頁設計（功能分類）
- 統計儀表板（關鍵指標）
- 麵包屑導航（明確位置）

#### 3. 視覺層次 (Visual Hierarchy)
```
設計手法：
大小對比 → 顏色區分 → 空間留白
```

**實踐方式：**
- 標題使用大字體
- 重要按鈕使用主色
- 卡片間距適當留白

#### 4. 一致性 (Consistency)
```
統一元素：
配色方案 → 按鈕樣式 → 間距規範
```

**實踐方式：**
- CSS 變數定義顏色
- 統一圓角半徑（8-12px）
- 固定間距系統（0.5rem倍數）

### 設計決策

#### **為什麼選擇分頁設計？**
- ❌ 原問題：單頁過長，需大量滾動
- ✅ 解決方案：分頁組織，專注單一任務
- 📈 效果：減少認知負擔，提升效率

#### **為什麼回饋系統放在獨立頁面？**
- 🎯 重要性：核心新功能，需突出展示
- 📝 複雜度：表單+列表，需足夠空間
- 👁️ 可見性：獨立頁面更易被發現

#### **為什麼使用星級評分？**
- 😊 直觀性：視覺化評分，一目了然
- 📊 標準化：1-5星統一標準，便於分析
- 🌍 通用性：國際通用符號，無語言障礙

## 🏗️ 技術架構

### 前端技術棧
```
HTML5 + CSS3 + JavaScript (ES6+)
├─ Bootstrap 5.3.0（UI框架）
├─ Bootstrap Icons（圖示庫）
└─ Fetch API（非同步請求）
```

### 後端技術棧
```
Spring Boot 3.2.0
├─ Spring Data JPA（資料存取）
├─ H2 Database（記憶體資料庫）
├─ Hibernate（ORM框架）
└─ RESTful API（介面設計）
```

### 資料模型
```
Station (車站)
├─ 基本資訊（代碼、名稱、路線）
├─ 位置資訊（地址、座標）
├─ 設施資訊（電梯、手扶梯等）
└─ 營運資訊（班次、運量）

Feedback (回饋)
├─ 乘客資訊（姓名、Email）
├─ 評分與分類
├─ 回饋內容
├─ 關聯車站（外鍵）
└─ 處理狀態與回覆
```
